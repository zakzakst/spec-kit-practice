## 1. ウォーターフォールの排除

**影響: クリティカル**

ウォーターフォールは最も重大なパフォーマンスの原因です。各逐次的な await は完全なネットワーク遅延を追加します。これらを排除することで最大の改善が得られます。

### 1.1 非同期フラグの前に安価な条件をチェックする

**影響: 高（安価な同期ガードが既に失敗する場合に不要な非同期処理を避ける）**

分岐がフラグやリモート値に対して `await` を使い、かつ安価な同期条件（ローカル props、リクエストメタデータ、既にロード済みの状態など）を必要とする場合、まず安価な条件を評価してください。そうしないと、合成条件が決して真にならない場合でも非同期呼び出しのコストを支払うことになります。

（以下、元の英語ドキュメントにあるコード例や詳細な説明はそのまま残し、コードブロックは変更しません。）

### 1.2 必要になるまで await を遅延する

**影響: 高（不要なコードパスでの待ちを避ける）**

`await` 操作を、それが実際に使われるブランチの内部へ移動させてください。これにより、必要のないコードパスで待機してしまうことを防ぎます。

### 1.3 依存ベースの並列化

**影響: クリティカル（2〜10× 改善）**

部分的に依存する操作については、`better-all` のような手法を使って並列性を最大化してください。これにより各タスクを可能な限り早く開始できます。

### 1.4 API ルートでのウォーターフォール連鎖を防ぐ

**影響: クリティカル（2〜10× 改善）**

API ルートや Server Actions では、独立した操作をすぐに開始してください（await しなくてもよい場合でも）。

### 1.5 独立操作に対する Promise.all() の活用

**影響: クリティカル（2〜10× 改善）**

依存関係のない非同期操作は `Promise.all()` で同時に実行し、ラウンドトリップ数を減らしてください。

### 1.6 戦略的な Suspense 境界の配置

**影響: 高（初期描画を高速化）**

非同期コンポーネントでデータを await してから JSX を返すのではなく、Suspense 境界を配置してラッパー UI を素早く表示し、データを読み込みながら部分的に表示することを検討してください。

## 2. バンドルサイズ最適化

**影響: クリティカル**

初期バンドルサイズを減らすことは、Time to Interactive（TTI）や Largest Contentful Paint（LCP）の改善に直結します。

### 2.1 バレルファイルのインポートを避ける

**影響: クリティカル（200〜800ms のインポートコスト、ビルド遅延）**

バレルファイルからのインポートは避け、必要なソースファイルから直接インポートしてください。多くのアイコンやコンポーネントライブラリはエントリーファイルに多数の re-export を含み、インポート時に大量のモジュールを読み込むことになります。

（以下は原文の説明・例のままです）

### 2.2 条件付きモジュール読み込み

**影響: 高（必要なときだけ大きなデータやモジュールを読み込む）**

機能が有効化されたときにのみ大きなモジュールを遅延読み込みするなど、条件付きで読み込むパターンを使ってください。

### 2.3 重要でないサードパーティライブラリを遅延読み込みする

**影響: 中（ハイドレーション後に読み込む）**

Analytics、ロギング、エラートラッキングなどの重要でないライブラリはハイドレーション後に読み込むことで初期バンドルを小さく保てます。

### 2.4 重いコンポーネントの動的インポート

**影響: クリティカル（TTI と LCP に直接影響）**

初期レンダリングで不要な大きなコンポーネントは `next/dynamic` 等で遅延ロードしてください。

### 2.5 ユーザー意図に基づくプリロード

**影響: 中（体感レイテンシの削減）**

ホバーやフォーカスで重いバンドルをプリロードするなど、ユーザーの意図に基づいて事前読み込みする手法を検討してください。

## 3. サーバーサイドパフォーマンス

**影響: 高**

サーバーサイドレンダリングとデータ取得の最適化は、サーバー側のウォーターフォールを排除し応答時間を短縮します。

### 3.1 API ルートと同様に Server Actions を認証する

**影響: クリティカル（サーバー側のミューテーションへの不正アクセスを防ぐ）**

`"use server"` を使う Server Actions はパブリックエンドポイントとして扱われるため、各アクション内で認証と認可を必ず行ってください。ミドルウェアやレイアウトガードだけに頼らないこと。

（原文のコード例と注意点はそのまま保持されています）

### 3.2 RSC プロップでの重複シリアライズを避ける

**影響: 低（ネットワークペイロードの削減）**

RSC→クライアントのシリアライズはオブジェクト参照で重複排除を行います。値ではなく参照が同じ場合のみ一度だけシリアライズされるため、サーバー側での配列やオブジェクトの変換は避け、クライアント側で行ってください。
